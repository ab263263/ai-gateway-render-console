# 号池 / 中转池控制台迭代规格（阶段一）

## 背景
当前 ai-gateway 已经具备：
- 多中转平台接入
- 远程模型抓取与导入
- 平台级聊天测试
- 虚拟模型 + 轮询

但产品形态还偏工程化，缺少类似 NewAPI 的“渠道池 / 号池控制台”体验。用户希望把它升级成专门管理第三方中转、多个 Key、不同模型可用性与轮询池的可视化后台。

## 产品定位
第一阶段将本项目正式定位为：

**面向 OpenAI 兼容中转的异步并发号池控制台**

它不是普通 AI Gateway 管理后台，而是用于：
- 接入多个第三方 OpenAI 兼容中转
- 管理不同 Key 当前支持哪些模型
- 统一做轮询、负载均衡、组池
- 供多个 agent / 多代理并发请求调用

## 本阶段目标
将当前测试能力与虚拟模型配置，升级为“号池 / 中转池控制台”的第一版。

## 关键架构约束

### 1. 上游协议范围
- 第一阶段主支持 **OpenAI Compatible** 中转平台
- 重点兼容：`/v1/models`、`/v1/chat/completions`
- 同时保留并持续支持 **Anthropic Messages API**：`/v1/messages`
- 也就是说，号池控制台的主场景是 OpenAI 兼容中转号池，但网关协议层必须继续同时兼容 OpenAI SDK 与 Anthropic SDK
- 前端巡检/测试结果后续也要能区分：
  - OpenAI chat 路径探测
  - Anthropic messages 路径探测
- Anthropic 等其他协议保留兼容能力，但第一阶段产品化设计仍以 OpenAI 兼容中转池为中心


### 2. 并发与多代理要求
- 号池必须支持异步并发请求
- 支持多 agent / 多代理同时调用
- 平台巡检与正式流量逻辑要隔离，避免后台测试影响正式转发
- 后续需预留：
  - 平台并发上限
  - 熔断 / 摘除
  - 健康评分
  - 自动恢复

### 3. 运行时状态管理
后续实现需支持跟踪：
- 某平台当前可用性
- 某模型最近成功率 / 错误率
- 平均延迟
- 是否限流 / cooldown / 无通道
- 请求模型与实际命中模型是否不一致

## 阶段一交付范围

### 1. 平台巡检 / 分层探测
对每个平台支持以下探测层：
- 基础连通性
- 鉴权检测
- `/models` 列表检测
- 指定模型实际调用检测
- 实际命中模型识别

### 2. 错误分类
测试结果需要结构化分类，而不是只返回一句 message。
建议类别：
- auth_error
- rate_limit
- cooldown
- model_not_found
- no_available_channel
- empty_model_list
- timeout
- network_error
- platform_compat_issue
- mapped_model_mismatch

### 3. 结构化结果表
在前端展示统一结果表，至少包含：
- 平台
- 请求模型
- 实际命中模型
- `/models` 是否成功
- `chat` 是否成功
- HTTP 状态
- 延迟
- 错误分类
- 错误详情

### 4. 批量巡检
支持：
- 单个模型测试
- 当前平台全部模型测试
- 批量结果汇总

### 5. 快速组池（第一版）
在虚拟模型页加入更直观的“快速组池”入口：
- 选择多个平台/模型
- 一键创建虚拟模型
- 选择负载策略（默认 RoundRobin）

## 验收标准
1. 平台级测试接口返回结构化结果，不只是 success/message
2. 前端能看出“请求模型”和“实际命中模型”的差异
3. 平台可批量测试全部远程模型并展示表格结果
4. 后台可更直观地把多个中转/不同模型组到一个轮询池
5. 设计上明确支持多 agent 并发调用，不再按单用户手工测试后台思路实现

## 约束
- 优先复用现有平台、模型、虚拟模型数据结构
- 先做第一版产品化，不一次性做到 NewAPI 完整复杂度
- 每次改动、部署、恢复都同步更新 `.workbuddy/memory/` 和 `ai-gateway/DEVLOG.md`
